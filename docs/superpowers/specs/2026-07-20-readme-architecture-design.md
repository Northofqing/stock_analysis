# 根 README 项目定位与架构重写设计

## 状态与依据

- 日期：2026-07-20
- 状态：已获用户直接实施授权；无需逐段确认。
- 依据：仓库当前 `master`、`Cargo.toml` target、`src/lib.rs` 模块树、生产入口、
  `AGENTS.md` 数据红线、`docs/business_rules.md` 与 v16.x 当前合同。

## 目标

让第一次进入仓库的开发者或运行人员在一个页面内准确回答：

1. 项目解决什么问题，不解决什么问题；
2. 真实数据如何经过分析、决策、风控、纸面执行、审计和通知；
3. 各核心目录的职责和依赖方向；
4. 当前能安全运行哪些入口；
5. 真实账户数据、纸面交易与尚未接入的券商实盘之间有什么边界；
6. 合并和发布必须通过哪些门禁。

## 方案选择

采用“用途优先 + 一张端到端架构图 + 分层职责 + 操作入口 + 安全合同”的 README。
不采用逐文件百科，因为它会掩盖主数据流且容易过时；不采用纯运维手册，因为项目定位和
架构关系会被命令列表淹没。

## README 信息架构

1. 项目标题、定位与当前成熟度
2. 能力与非目标
3. Mermaid 端到端数据流
4. 六层架构职责表
5. 关键数据流：研究/监控、纸面交易、事件投递、真实账户快照
6. 可执行入口与适用场景
7. 数据、资金、测试隔离和审计安全合同
8. 环境准备与最小启动步骤
9. Gate A–D、测试与覆盖率命令
10. 当前限制和权威文档入口

## 架构表达

README 只描述当前可执行合同：

```text
真实公共数据/本地账户证据
  -> provider 与完整批次校验
  -> SQLite/JSONL 审计存储
  -> pipeline/opportunity/decision
  -> risk/order_safety
  -> paper trading 或受治理的通知
  -> review/performance/replay
```

模块按职责归为六层：应用入口、数据接入、持久化、研究决策、风险与纸面执行、事件与通知。
具体模块名必须来自当前模块树，不使用历史版本的计划名冒充已实现模块。

## 真实性和措辞约束

- 不写固定源码行数、模块数、测试数或推送条数。
- 不把历史版本号称为“最新进展”；当前状态以主分支和权威审计文档为准。
- 不宣称 IB/TQS/QMT 已接入；未支持的 broker source 必须显式失败。
- 不宣称系统会自动实盘下单；当前交易闭环是受安全门约束的纸面交易与审计。
- 不展示真实账户值、证券身份、截图路径或本地私有证据路径。
- 缺失字段描述为 `NULL`/不可用/显式错误，不描述为 0 或默认值。
- 周末不要求产生行情；日线新鲜度按交易日历计算。

## 失败模式

| 风险 | 处理 |
| --- | --- |
| README 命令与真实 target 不一致 | 只列 `cargo metadata` 可解析且用途明确的入口 |
| 历史设计与当前实现冲突 | README 链接历史文档，但以当前合同/业务规则为权威 |
| 用户把纸面交易误解为券商实盘 | 在开头、架构图和限制章节重复明确边界 |
| 示例诱发真实通知或真实下单 | 默认示例只用只读/测试安全命令；生产模式要求显式配置 |
| 文档再次因数量变化而失真 | 避免硬编码易变计数，只保留门禁阈值与本次可复验结果 |

## 验证

- 对照 `cargo metadata --no-deps` 核验二进制名称。
- 对照 `src/lib.rs` 与生产调用链核验架构模块。
- 对照 `AGENTS.md`、BR-051/084/086/092/103/108/111/130/134 核验安全表述。
- 执行 Markdown 差异检查、格式、严格 Clippy、全测试、合规、release build 和 Gate D 覆盖率。
- 固定提交后进行 Standards 与 Spec 双轴独立审计。

## 回滚

README、设计文档和确定性测试修复使用可审计提交，经 PR 合入。回滚使用
`git revert <merge-commit>`；不得在回滚文档时恢复 mock、成本价或零值回退。
