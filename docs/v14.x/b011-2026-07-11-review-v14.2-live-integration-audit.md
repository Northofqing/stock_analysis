## 0. 一句话结论

**v14.2 七层推送架构是一个"影子层"：它在旁边观察、往数据库写记录，但真正的推送走的仍是 v13 老路径；而且它写进数据库的数据是错的。**

1058 个单元测试全绿掩盖了这一点 —— 因为**没有一个测试检查集成**。这正是 CLAUDE.md「完成三条件」硬规则要防的那类幻觉：模块层 ✅ + 测试 ✅，但实机行为 ❌。

| 层 | 文档 §3.x 声称职责 | 实机真相 | 判定 |
|---|---|---|---|
| L1 信号源 | SignalEvent 生成 | 在线，event_id 正常生成 | ✅ |
| L2 模板注册 | 36+ 模板注册表 | **只有 316 行类型定义，无注册表** | ❌ 骨架 |
| L3 渲染层 | 文档 §3.3 完整定义 | **`src/push_l3/` 目录不存在** | ❌ 缺失 |
| L4 仲裁/去重 | dedup 闭环 | **stub，无条件返回 Pushed** | ❌ 假的 |
| L5 治理 | quiet/frozen/data_mode | 在线，但 governance_decision 恒 Approve | ⚠️ 未验伪 |
| L6 投递 | Sink 路由 | 在线，但只有 ConsoleSink | ⚠️ |
| L7 分析 | 推送效果归因 | 在线，**但记录假数据** | ❌ 有害 |

## 1. P0 级问题（必须先修，否则越跑越错）

### P0-1. L7 Analytics 在系统性地记录假数据 —— `sink_name` 与真实投递渠道不符

**实证（数据库真实行）**：

```
$ sqlite3 -header data/push_analytics.db "SELECT * FROM push_analytics LIMIT 2;"
id|event_id|template_id|...|governance_decision|pushed|rendered_len|sink_name|user_id
1 |11158d77682c6971|daily_report  |...|Approve|1|1006|wechat |default
2 |32fff41904c72780|holding_event |...|Approve|1|11  |console|default
```

**同一时刻实机日志显示真实投递渠道是飞书**：

```
[23:26:00 INFO] [飞书] 推送成功 | to=oc_4bca5d870fd5ff3352795a674194d5b0 |
                send ok (feishu): message_id=bae74fcc-1040-4185-9d66-c8a9259e2ee4
```

**根因**：`src/bin/monitor/notify.rs:493` 硬编码 `"wechat"` 字符串：

```rust
let outcome = v14_adapter::v14_dispatch(kind, text, "wechat");
```

`sink_name` 从来不是"实际投递到哪"，而是一个写死的字面量。ConsoleSink 的记录则写成 `console`。

**危害（量化视角）**：`push_analytics` 表是未来做推送效果归因的唯一数据源 —— 哪个渠道到达率高、哪类模板被 governance deny、哪些推送带来了实际交易动作。**基于这张表得出的任何结论都是错的**。对量化系统而言，拿错误数据做决策比没有数据更危险：没有数据你会去找数据，有错误数据你会直接下结论。

**处置决策（待拍板）**：
1. 修 `notify.rs:493`，把真实 sink 从投递层回传，而不是入口硬编码；
2. 已落库的历史行 —— **清空重来**还是**标记为不可信**？（建议清空：数据量小，且混着假数据的表比空表更危险）

### P0-2. L4 Dispatcher 是 stub —— 去重根本没生效

**实证（实机日志）**：

```
[23:27:51 INFO] [dispatcher] W4.1 stub: event_id=1e56674266dd93db source=HoldingHealth kind=daily_report
[23:28:04 INFO] [dispatcher] W4.1 stub: event_id=90e0145a40ce1d3c source=HoldingHealth kind=review_market
```

**源码**（`src/push_l4/dispatcher.rs:95-103`）：

```rust
// W4.1 暂跳过, 直接 Pushed (待 W4.2 接入 template 后再实现)
log::info!("[dispatcher] W4.1 stub: event_id={} ...", ...);
self.stats.dispatched += 1;
DispatchOutcome::Pushed          // ← 无条件放行
```

**危害**：文档 §3.4 把 dedup 闭环的职责给了 L4，`notify.rs` 的注释也写着「冷却由各 dispatcher 自己 memo (D-01, P-04 等)」。但 L4 无条件放行 —— **真正的去重仍然全靠老的 `push_governor_v2` cooldown 兜底**。

这比"没有 L4"更糟：一个 stub 在日志里持续打印 `[dispatcher]`，让人以为去重已经迁移到新架构了。**你以为换了新刹车，其实还是老刹车，新刹车是个模型。**

**处置决策（待拍板）**：要么实现 W4.2，要么把 stub 摘掉并在日志里明确「dedup 仍由 v13 push_governor_v2 负责」。**留着它比没有更危险。**

## 2. P1 级问题

### P1-1. 推送漏斗零可观测性 —— 32 条信号如何变成 0 条推送，无人知晓

**实证（实机日志，连续 3 轮完全一致）**：

```
[22:56:42 INFO] [公告] 2026-07-11 获取 100 条
[22:56:42 INFO] [公告] 过滤后 32 条需告警
        ← 然后什么都没有。没有推送，没有"被去重"，没有"被过滤"，没有任何日志。
[22:58:43 INFO] [公告] 过滤后 32 条需告警     ← 下一轮，还是 32 条
[23:00:43 INFO] [公告] 过滤后 32 条需告警     ← 再下一轮，还是 32 条
```

日志出处 `src/data_provider/announcement.rs:526`。这 32 条之后经过 `nm.process_announcements()`（`main.rs:3582`）+ 状态机 `sm.process(e)`（`main.rs:3587`）两道过滤，全部归零。

**这不是 bug（自选股 gating 是设计如此），但是可观测性缺口。**

**危害（量化视角）**：实盘系统里，信号从产生到被丢弃的**每一跳都必须可追溯**。否则你永远分不清三种情况：
- 今天真的没信号（市场安静）
- 过滤器把信号吃了（阈值配错）
- 上游数据坏了（数据源静默降级）

这三者在当前日志里**长得一模一样**。而 AGENTS.md 明确要求「Missing data → log warning, don't silently fill」—— 漏斗静默归零是同一类问题的变体。

**建议**：每一级过滤器输出 `进N条 → 出M条 | 丢弃原因分布`。

---

### P1-2. 四条代码路径共同决定"这条推送要不要发"

`src/bin/monitor/notify.rs` 现存三代接口 + 一条旁路：

| 路径 | 签名 | 调用点 | 状态 |
|---|---|---|---|
| `push_governor` | `→ bool` | **36 处 caller** | `notify.rs:526` 兼容层 |
| `push_governor_v2` | `(kind, code) → bool` | 冷却兜底主力 | `notify.rs:548` |
| `push_governor_v3` | `→ PushOutcome` | 新代码推荐 | `notify.rs:531` |
| `V10_DRY_RUN_PUSH=1` | 环境变量旁路 | 绕过 v14.2 直通 v13 `push_wechat` | `notify.rs:515` |

全仓 `push_governor` 相关引用 **116 处**。四条路径并存 = 事故温床。文档 Phase 3 DoD 要求「dispatch 入口单事件源（v13 push_governor 已 `#[deprecated]`）」—— 尚未开始。

## 7. 一句话总结

**基础设施健康，推送层是个精致的空壳。**

v14.2 写了七层里的六层代码、1058 个测试全绿 —— 但 L3 不存在、L4 是 stub、L2 只有类型、L7 在记假数据，而真实推送仍走 v13 老路。

**最危险的不是"没做完"，而是"看起来做完了"**：`[dispatcher]` 在日志里持续打印、`push_analytics.db` 在持续落库 —— 这些信号都在告诉你新架构在工作，但它没有。

这正是 CLAUDE.md「完成三条件」硬规则存在的理由，也是这条规则第一次被实机验证抓到现行。
