# v18.0 Quant Platform Closure — Implementation Roadmap

> **For agentic workers:** Execute one workstream at a time. Each workstream must produce a reviewable design update, tests, Gate evidence, and a small PR. Do not begin the next workstream when its predecessor's Gate is open.

**Goal:** Deliver a reproducible, cost-aware, risk-controlled paper-trading loop and define the evidence required before any controlled live execution work begins.

**Architecture:** Add authoritative data/decision/event contracts around existing modules instead of replacing the monitor. Data health and provenance gate every decision; a unified risk policy gates paper orders; a durable event-backed ledger and review loop provide falsification. SQLite is an operational projection only; a WORM-retained immutable audit journal is the audit system of record. Live broker routing remains disabled and is a future separate project.

**Tech Stack:** Rust 2021, Tokio, Diesel/SQLite, existing event/push layers, existing A-share risk/market-data modules.

---

## Workstream order

| Order | Workstream | Depends on | Gate |
| --- | --- | --- | --- |
| 1 | Data contracts and health authority | none | no default/fake decision input; stale thresholds enforced |
| 2 | Research/model registry and promotion evidence | 1 | point-in-time and net-cost evidence required |
| 3 | Decision, portfolio, and unified risk policy | 1, 2 | every paper action has a durable decision and risk result |
| 4 | Paper order/fill/ledger/reconciliation | 1, 3 | paper ledger invariant/reconciliation tests pass |
| 5 | Outcome attribution and operating review | 2, 4 | model monitoring and falsification reports trace to decisions |
| 6 | Live-execution readiness review | 1–5 | design-only Gate L decision; no broker order code in this workstream |

## Workstream 1: Data contracts and health authority

**Likely files:** `src/data_provider/service.rs`, `src/monitor/data_quality.rs`, `src/monitor/data_mode.rs`, `src/bin/monitor/freshness.rs`, `src/bin/monitor/market_data.rs`, focused unit/integration tests, `docs/business_rules.md` if filtering/dedup/limits are introduced.

1. Write tests that prove provider failure yields the sole typed `Unavailable` result and cannot become a numeric factor/quote input.
2. Introduce typed data envelopes with source time, receive time, adjustment basis, quality status, and validation evidence.
3. Replace decision-path `unwrap_or_default`/empty fallbacks with explicit errors or unavailable states; preserve logging as additional evidence, never as the only behavior.
4. Build one `DataHealthSnapshot` from real timestamps. Enforce 5-second quote and 30-second account requirements for action paths.
5. Delete or quarantine production constructors that synthesize `Full`; provide explicitly test-scoped fixtures only.
6. Register any new filtering/dedup/rate-limit rule, run focused tests, then run full Gate B/C commands.

**Acceptance tests:** stale quote at 6 seconds rejects an increasing order; missing money-flow cannot equal zero flow; duplicate/out-of-order bar is invalid; source conflict produces `Conflicted`; test code cannot construct a live order path.

## Workstream 2: Research/model registry and promotion evidence

**Likely files:** `src/pipeline/backtest_runner.rs`, `src/strategy/core.rs`, `src/database/factor_snapshot.rs`, new `src/research/` boundary, new persistence migration, research tests.

1. Add a versioned research-run/model record containing code revision, universe, calendar, data snapshot identifiers, adjustment basis, config hash, and cost model version.
2. Make point-in-time factor availability mandatory for promotable backtests; label legacy slice-end-factor results exploratory.
3. Add walk-forward/out-of-sample report generation with gross/net return, turnover, capacity, drawdown, exposure, and fill/cost assumptions.
4. Add an independent review record with limitations, approval/rejection, expiry, and monitoring thresholds.
5. Verify a modified data or config snapshot changes the research identity and cannot overwrite an approved result.

**Acceptance tests:** a look-ahead-compatible run cannot receive paper-promotion status; a missing snapshot fails explicitly; net return changes when cost assumptions change; approved model record is immutable.

## Workstream 3: Decision, portfolio, and unified risk policy

**Likely files:** `src/decision/`, `src/risk/`, `src/trading/risk_adapter.rs`, `src/portfolio/`, new `src/decision/record.rs`, new persistence migration, `docs/business_rules.md`.

1. Define `DecisionRecord` and `RiskDecisionRecord` with stable IDs and all provenance/config/risk references.
2. Make portfolio construction use expected net value, liquidity/capacity, turnover, cash, concentration, exposure, and drawdown constraints.
3. Consolidate current paper risk checks into one policy interface used for every paper order.
4. Require fresh authoritative data health, account isolation, A-share hard limits, duplicate-ID rule, and confirmation rule before approval.
5. Persist accept and reject results before any paper-execution side effect.

**Acceptance tests:** the same request receives the same decision ID behavior; stale/missing/conflicted data rejects; a 500,000 RMB threshold path requires second confirmation; paper and future broker adapters share the same policy fixture suite.

## Workstream 4: Paper order, fill, ledger, and reconciliation

**Likely files:** `src/trading/paper_trade.rs`, `src/trading/paper_engine.rs`, `src/database/`, new paper-order/fill/ledger modules and migration, `src/review/` tests.

1. Preserve the current paper/live isolation while moving from a one-shot simulated record to parent orders, child fills, lifecycle states, and explicit no-fill/partial-fill outcomes.
2. Version the fill and cost assumptions; record quote/data snapshot evidence used for every fill.
3. Build an event-backed paper ledger for cash, positions, fees, and P&L; SQLite summary rows are rebuildable projections, not the authoritative audit record.
4. Before accepting every critical paper-loop event, obtain a WORM/Object-Lock audit receipt with five-year retention. Canonical payloads form a hash chain; seal and independently retain a daily signed Merkle root. Archive failure is fail-closed for order submission.
5. Implement daily invariant and reconciliation jobs. Verify every immutable receipt and daily root, detect archive-orphan records, and stop new paper orders on a mismatch.
6. Add a retention-policy probe and restore/read drill to Gate P evidence; a SQLite append-only table or hash chain without immutable retention is insufficient.
7. Migrate old paper records read-only; do not reinterpret them as new-model fills.

**Acceptance tests:** limit-up buy becomes a durable not-filled event; T+1 prevents invalid sell; partial fills reconcile; retry does not duplicate; ledger replay recreates cash/position exactly; an archive receipt with fewer than five-year lock fails; missing audit receipt blocks submission; mismatch blocks the next order.

## Workstream 5: Outcome attribution and operating review

**Likely files:** `src/review/`, `src/performance/`, `src/database/execution_tracking.rs` replacement path, `src/bin/monitor/` adapters, report tests.

1. Replace in-memory execution statistics with event-backed outcomes keyed by `decision_id`.
2. Report expected versus realized cost, fill rate, implementation shortfall, net performance, calibration, exposure, drawdown, and results by market regime.
3. Add model-monitoring thresholds and a durable restriction/promotion decision; review cannot directly edit a production configuration.
4. Keep notifications as renderings of already-durable review results.

**Acceptance tests:** a review report can traverse outcome → fill → order → risk decision → decision → inputs; missing evidence is shown as missing, not filled; monitoring breach restricts paper promotion.

## Workstream 6: Live-execution readiness review

**Files:** a new v18 design/review document, runbooks, threat/failure tests, and no production broker write path.

1. Audit Gate P evidence over an agreed paper period.
2. Specify a broker adapter contract with durable state machine, broker queries, reconciliation, kill switch, authorization, and two-person confirmation.
3. Tabletop-test ambiguous acknowledgement, broker outage, restart, duplicate submit, cash/position mismatch, and kill-switch recovery.
4. Obtain explicit approval for a separate live-execution design; without it, remain in paper mode.

## Global validation and rollback

For each workstream run, at minimum:

```bash
cargo fmt --check
cargo clippy -- -D warnings
cargo test
bash tools/compliance/check.sh
git diff --check
```

Rollback is per small PR using `git revert <commit>`. Database/event migrations require forward-safe compatibility and a documented restore/replay procedure; never delete audit or paper-ledger records to roll back.
