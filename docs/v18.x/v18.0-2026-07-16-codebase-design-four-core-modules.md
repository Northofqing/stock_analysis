## 0. 设计边界

四个模块是新业务语义的深模块（deep modules）：调用方只看到少量稳定接口，数据源细节、SQLite 投影、异步归档、填单模型与指标公式留在模块实现内部。它们通过明确的 seam 相连，不能让 `main.rs`、推送模板或 UI 直接拼接领域状态。

```text
Existing source adapters ─┐
                           ▼
                  ① DataHealthSnapshot
                           │ DataEnvelope + snapshot_id
existing strategy ────────┼───────────────┐
portfolio/risk context ───┘               ▼
                                     ② DecisionRecord
                                              │ approved paper intent
                                              ▼
                               ③ Paper Order / Fill / Ledger
                                              │ immutable event references
                                              ▼
                            ④ Attribution / ModelChangeProposal
                                              │ reviewed proposal only
                                              └──── Champion/Challenger next paper run
```

共同不变量：

- `correlation_id` 贯穿一次评估；`decision_id` 是之后所有订单、成交、复盘的不可变连接键。
- 任何会影响行动的输入都必须有 `DataEnvelope`、时间语义和来源；`Option::None` 不能被解释为零、空集合或“正常”。
- 所有接受或拒绝的关键状态改变都先取得 `AuditReceipt`；本地 SQLite 仅为可重建投影。
- 纸面账本与真实账户真相严格分离；v18 不调用任何 broker write adapter。

## 1. 共同值对象与审计 seam

这不是第五个业务模块，而是四个模块共享的窄基础接口。接口留在 `src/audit/`，适配器留在 `src/audit/adapters/`；各模块只依赖 trait，不依赖对象存储或 Diesel。

```rust
pub type CorrelationId = Uuid;
pub type DecisionId = Uuid;
pub type AuditEventId = Uuid;

pub struct AuditReceipt {
    pub event_id: AuditEventId,
    pub payload_hash: Sha256Hash,
    pub chain_hash: Sha256Hash,
    pub archive_object_version: String,
    pub archived_at: DateTime<Utc>,
    pub retention_until: DateTime<Utc>,
}

pub trait AuditJournal: Send + Sync {
    fn append_critical(
        &self,
        event: CanonicalAuditEvent,
    ) -> Result<AuditReceipt, AuditError>;
    fn verify_day(&self, trading_day: NaiveDate) -> Result<AuditVerification, AuditError>;
    fn retention_status(&self) -> Result<RetentionEvidence, AuditError>;
}
```

`CanonicalAuditEvent` 使用固定 schema version、稳定字段排序和 UTC 时间；不得把日志字符串作为审计载荷。每个事件记录 `event_type`、主体 ID、`correlation_id`、前序哈希、输入快照引用、操作者/服务身份与拒绝原因（若有）。敏感凭据不写入载荷。

最小适配器的写入顺序是：生成 canonical event → 写入带 WORM/Object-Lock 的不可变对象并取得版本化回执 → 在同一 SQLite transaction 写领域事件和回执 → 更新可重建投影。第二步失败则不接受纸面订单；第三步失败则不可变对象成为“归档孤儿”，日终 reconciliation 必须把它找出并人工归并或以纠正事件关闭。每日将事件链叶子计算签名 Merkle root，写入另一份受保留锁保护的对象。

保留策略的实现配置不硬编码在 Rust：部署配置必须声明合规锁模式与 `retention_until >= event_time + 5 years`。`AuditJournal` 启动自检和 Gate P 都读取远端对象属性；本地文件、普通 S3 bucket 或仅 SQLite hash chain 均不能返回合格 `RetentionEvidence`。

## 2. 模块一：Data Contract Gate / DataHealthSnapshot

### 职责与接口

模块位置建议为 `src/data_contract/`。它将现有 provider、`monitor/data_quality.rs`、`monitor/data_mode.rs` 与 freshness 代码收敛成一个行动前的数据事实，不负责拉取 HTTP，也不决定投资信号。

```rust
pub enum Availability {
    Available,
    Unavailable { reason: UnavailableReason, retryable: bool },
    Stale { observed_age: Duration, allowed_age: Duration },
    Invalid { rule_id: String, evidence: ValidationEvidence },
    Conflicted { observations: Vec<ObservationRef> },
}

pub struct DataEnvelope<T> {
    pub value: Option<T>,
    pub source: SourceProvenance,
    pub as_of: DateTime<Utc>,
    pub received_at: DateTime<Utc>,
    pub schema_version: String,
    pub adjustment_basis: Option<AdjustmentBasis>,
    pub availability: Availability,
}

pub struct DataHealthSnapshot {
    pub snapshot_id: Uuid,
    pub generated_at: DateTime<Utc>,
    pub quote: CapabilityHealth,
    pub account: CapabilityHealth,
    pub daily_history: CapabilityHealth,
    pub required_features: BTreeMap<FeatureName, CapabilityHealth>,
}

pub trait DataHealthAuthority {
    fn assess(&self, input: DataHealthInput, now: DateTime<Utc>)
        -> Result<DataHealthSnapshot, DataContractError>;
    fn check(&self, snapshot: &DataHealthSnapshot, intent: ActionIntent)
        -> Result<(), DataHealthRejection>;
}
```

不变量是：只有 `Available` 可以携带 `Some(value)`；其余四种状态必须携带 `None`，并带可诊断原因。`Unavailable` 是唯一的 provider/网络/授权/无结果状态，不再出现 `Missing`、空对象或 `unwrap_or_default()` 三种并存的语义。字段可选与源不可用要分开表示：例如财报“该季度没有该字段”是 `Available` 载荷中的 `None` 加字段缺失告警；财报源失败是 `Unavailable`。

`check` 采用 BR-038：开仓、加仓、调仓要求所有被声明为 required 的能力均 `Available` 且未过期。即时行情阈值 5 秒、账户 30 秒、日线按交易日规则来自单一 `FreshnessPolicy`。减仓/平仓不在此模块暗自放行；它仍由调用方传入的版本化 `ActionIntent` 和风险策略决定。任何拒绝都有 `DataHealthRejection` 与审计事件。

### 适配与持久化

| 现有形态 | 适配器责任 | 不应保留的行为 |
|---|---|---|
| `data_provider/fallback.rs` | 将已验证的多源日线结果包装为 `DataEnvelope<DailyBars>`，保留全部源观察引用 | “最后一个错误”变为空数据继续计算 |
| `monitor/data_quality.rs` | 产出 `ValidationEvidence`（价格、连续性、除权、重复） | 仅 log 后仍把数据交给策略 |
| `data_provider/service.rs` | `get_financials`、`get_money_flow`、`get_intraday_shape` 改为返回 envelope/result | `unwrap_or_default` 形成可行动的零值/空值 |
| `monitor/data_mode.rs`、`bin/monitor/freshness.rs` | 改为 `DataHealthAuthority` 的输入适配器 | 生产路径直接构造 `DataMode::Full` |

持久化表只存快照元数据、来源引用、状态和审计回执，不复制大行情载荷：`data_health_snapshots`、`data_capability_health`、`data_incidents`。原始行情仍由现有数据表/数据源版本保存。`snapshot_id` 必须能回指实际 source/as-of/schema，而不是只保留“健康/不健康”布尔值。

### 失败语义、测试 seam 与最小验收

`DataHealthAuthority` 的纯 `assess`/`check` seam 用固定时钟测试。必须覆盖：6 秒行情拒绝开仓、31 秒账户拒绝加仓、provider 超时生成 `Unavailable` 而非零资金流、价格≤0/重复时间生成 `Invalid`、两源冲突生成 `Conflicted`、日线跨非交易日仍合法。集成测试确认 `DataHealthRejection` 不能被 `RiskPolicy` 或 `PaperExecution` 绕过。

## 3. 模块二：DecisionRecord / CandidateSnapshot

### 职责与接口

模块位置建议为 `src/decision/`，只负责把一次完整候选评估与组合意图固化；它不发订单、不发送推送，也不对候选集做隐式截断。候选集完整性是复盘“当时为什么没买另一只”的前提。

```rust
pub struct CandidateBatch {
    pub candidate_batch_id: Uuid,
    pub correlation_id: CorrelationId,
    pub strategy_version: VersionRef,
    pub model_version: VersionRef,
    pub config_hash: Sha256Hash,
    pub universe_snapshot_id: Uuid,
    pub data_health_snapshot_id: Uuid,
    pub portfolio_revision: u64,
    pub evaluated_at: DateTime<Utc>,
    pub candidates: Vec<CandidateEvaluation>,
}

pub enum CandidateDisposition {
    Eligible,
    Rejected { rule_id: String, reason: String },
}

pub struct CandidateEvaluation {
    pub instrument: InstrumentId,
    pub rank: Option<u32>,
    pub disposition: CandidateDisposition,
    pub feature_snapshot_refs: Vec<SnapshotRef>,
    pub signal_evidence: SignalEvidence,
    pub expected_return: Decimal,
    pub uncertainty: Decimal,
    pub liquidity: LiquidityEstimate,
    pub expected_cost: CostEstimate,
}

pub trait DecisionService: Send + Sync {
    fn record(
        &self,
        batch: CandidateBatch,
        construction: PortfolioConstructionInput,
    ) -> Result<DecisionRecord, DecisionError>;
    fn get(&self, decision_id: DecisionId) -> Result<DecisionRecord, DecisionError>;
}
```

`record` 首先验证候选批次 key（strategy/model/config/universe/data-health/portfolio revision）和 BR-039。一模一样的提交返回已持久化的 `DecisionRecord`；同一批次不同载荷是完整性冲突，必须拒绝并审计，而非覆盖。组合构造可以在内部使用现有 `portfolio/*`、`strategy/core.rs`，但输出只能是目标仓位、差额、成本/容量/约束证据与 `DecisionOutcome::{Proposed, Rejected}`。所有被排除标的保留排除规则和原因。

`DecisionRecord` 一旦取得审计回执即不可变。新的行情、账户、模型或配置产生新 record；不能更新旧 record 的 score、预期收益或目标仓位。推送模块只能读取渲染它，纸面执行只能接收其 `ApprovedPaperIntent`，因此通知重试绝不会重做决策。

### 存储、适配和失败语义

表结构建议为 `candidate_batches`、`candidate_evaluations`、`decision_records`、`decision_constraints`，以 `decision_id`/`candidate_batch_id` 建唯一索引，审计回执单列存引用。大特征载荷存已有 snapshot 表或对象存储，只存 hash/ref；这保证可重放且避免把整个因子矩阵复制进 SQLite。

旧模块的 seam：`strategy/core.rs` 是 `SignalEvidence` adapter；`pipeline/backtest_runner.rs` 只产生已经通过 point-in-time 检查的 `SnapshotRef`；`portfolio/*` 是 `PortfolioConstructionInput` adapter；`push_l1`–`push_l7` 消费持久化后的 decision view。`main.rs` 只能组织这些 adapter，不能自行计算或造 `decision_id`。

必测项目：同输入重试只得到同一 ID；候选数量与 universe 快照一致；一个过滤/排序/容量限制没有 BR ID 时拒绝构造；任一需要特征为非 `Available` 时没有 `ApprovedPaperIntent`；缺少成本或流动性证据时结果是可解释 `Rejected`，不是假定零成本。

## 4. 模块三：Paper Order / Fill / Ledger

### 职责与状态机

模块位置建议为 `src/paper_ledger/`。它把现有 `trading/paper_trade.rs` 与 `paper_engine.rs` 的一次性模拟升级为可重放事件模型，但继续与 `portfolio/*`、真实账户和未来 broker adapter 物理隔离。

```text
Proposed → RiskApproved → Submitted → PartiallyFilled → Filled
                         │                │
                         ├─ Rejected      ├─ Cancelled
                         │                ├─ Expired / NotFilled
                         │                └─ ReconciliationBlocked
                         └─ AuditUnavailable (no order accepted)
```

`RiskApproved` 不是交易所状态，表示统一 `RiskPolicy` 已验证 BR-038、现有 A 股硬限制、账户命名空间、幂等及二次确认。`Submitted` 前必须已有决策与风险两份审计回执；每个 fill 也独立有数据快照、成本模型版本和审计回执。`ReconciliationBlocked` 是账本运行状态，不把历史订单改成“未发生”。

```rust
pub trait PaperExecution: Send + Sync {
    fn submit(
        &self,
        intent: ApprovedPaperIntent,
        quote: DataEnvelope<Quote>,
    ) -> Result<PaperOrderReceipt, PaperExecutionError>;
    fn process_market(
        &self,
        order_id: PaperOrderId,
        market: FillMarketSnapshot,
    ) -> Result<Vec<PaperEvent>, PaperExecutionError>;
    fn reconcile(&self, trading_day: NaiveDate)
        -> Result<ReconciliationReport, ReconciliationError>;
}

pub trait PaperLedgerReader: Send + Sync {
    fn portfolio_at(&self, as_of: DateTime<Utc>) -> Result<PaperPortfolio, LedgerError>;
    fn events_for(&self, decision_id: DecisionId) -> Result<Vec<PaperEvent>, LedgerError>;
}
```

`submit` 以 BR-040 的 `paper_order_key = decision_id + target_revision + side` 幂等。它先检查账本没有被阻塞、验证 quote envelope、调用统一风险策略，写入不可变审计事件，再创建订单。任何阶段的重试都返回已存在的 receipt 或清晰失败，不能多扣现金或多一笔虚拟仓位。

`process_market` 是唯一可创建 fill 的入口；它以明确版本化 `FillModel` 处理价格限制、T+1、参与率、盘口/成交量、佣金、印花税、滑点、部分成交及未成交。模型缺参数时输出 `NotFilled` 或错误，不得假设理想成交。`PaperLedger` 由事件重放得到现金、冻结资金、持仓、可卖数量、费用和已/未实现 P&L；汇总表可删除后重建。

### 存储、审计与对账

核心表为 `paper_orders`、`paper_events`、`paper_fills`、`paper_ledger_projection`、`paper_reconciliations`，每条关键事件存 `audit_event_id`/receipt ref。真实 `database/positions.rs` 只能作为风险中的真实账户输入，禁止写入 paper 结果。

`reconcile` 对同一交易日从 `paper_events` 重放，比较订单状态、现金、持仓、可卖量、费用和每个 `AuditReceipt` 的链/日根。任何差异、孤儿归档对象或缺少回执会写入 incident，并把账本状态置为 `ReconciliationBlocked`；下一笔 `submit` fail-closed。纠正通过新 `CorrectionRecorded` 事件完成，并保留原事件，不允许 UPDATE 汇总以“修平”。

最小测试 seam 是可注入 `FillModel`、`AuditJournal`、`Clock` 和只读 `MarketSnapshotStore`。验收至少包括：涨停买入 `NotFilled`、T+1 卖出拒绝、部分成交费用正确、同 key 重试无重复、重放等于投影、少于五年保留锁的回执拒绝订单、缺回执/对账差异阻断新订单。

## 5. 模块四：Attribution / ModelChangeProposal

### 职责与接口

模块位置建议为 `src/review/`（指标计算可置于 `src/performance/`），把“策略看起来有效”变成能复核、能证伪、不能越权改参数的复盘输出。它不生成行情、不写订单、不编辑 `.toml`。

```rust
pub struct ReviewScope {
    pub model_version: VersionRef,
    pub strategy_version: VersionRef,
    pub paper_book_id: PaperBookId,
    pub from: DateTime<Utc>,
    pub to: DateTime<Utc>,
    pub benchmark: BenchmarkRef,
    pub regime_definition: VersionRef,
}

pub trait AttributionService: Send + Sync {
    fn review(&self, scope: ReviewScope) -> Result<AttributionReport, ReviewError>;
    fn propose(
        &self,
        report_id: Uuid,
        policy: ProposalPolicy,
    ) -> Result<Vec<ModelChangeProposal>, ReviewError>;
}
```

`review` 只能沿 `decision_id → risk result → paper order → fills → ledger → market outcome` 链读取。它同时保留“入选”与“未入选”的候选集，按模型版本、持有周期、市场状态、行业/风格暴露、rank 桶、流动性桶和拒绝原因分组。每一笔建议至少有 T+1/T+5/T+20、退出时和最大不利/有利变动标签；不足观察窗口的标签保持 `Unavailable`，不补零。

报告的最小指标包括：毛/净 P&L、相对基准收益、命中率、收益分布、换手、胜率校准、预测与实际成本差、fill rate、implementation shortfall、参与率、集中度、行业/风格暴露、回撤及失败假设。收益归因必须分别呈现信号、组合构造、执行成本和市场 beta/风格暴露，不能将成本后的失败归为“模型预测错误”。

### 自我迭代的治理状态机

```text
AttributionReport
      │ evidence + sample-sufficiency policy
      ▼
ModelChangeProposal(Draft) → Reviewed → Shadow → PaperChallenger
                                  │                  │
                                  ├─ Rejected         ├─ Promoted (human approval)
                                  └─ NeedsMoreData    └─ Restricted / Retired
```

提案包含建议变更（模型、特征、风险预算或执行假设）、Champion/Challenger 版本、训练/验证数据边界、样本充分性策略版本、预期收益和风险影响、反事实对照、回滚版本及审批记录。它本身不写配置；从 `Promoted` 到生效必须由独立、带 PR 的版本化配置/模型发布流程完成，并引用本设计、BR-042 和阈值证明。

Champion/Challenger 不是用事后不同输入比较：二者消费相同的 universe、`DataHealthSnapshot`、行情时间、账户快照与 fill/cost 假设，各自维护 `paper_book_id`。因此可比较“若同一时刻按 Challenger 决策会怎样”，且不会污染 Champion 账本。晋升不依赖单一夏普/胜率：`SampleSufficiencyPolicy` 必须同时检查样本量与持有期成熟度、市场状态覆盖、成本后结果、尾部风险/回撤、数据可用率、多重比较说明和经济容量；任一缺失只允许 `NeedsMoreData`。

### 存储、适配和测试 seam

建议表：`outcome_labels`、`attribution_reports`、`attribution_slices`、`model_change_proposals`、`model_governance_events`。所有指标行保存输入 decision/fill/ledger/benchmark/regime 的版本与 hash。`database/execution_tracking.rs` 只能作为迁移数据来源，不能继续作为内存真相；现有 post-close/push 模块只能渲染已经持久化的 report。

最小 seam 是只读 `DecisionReader`、`PaperLedgerReader`、`MarketOutcomeReader`、`BenchmarkReader`、`RegimeClassifier` 与 `GovernanceRepository`。必测：相同输入得到相同 report hash；任一链路缺失显示 `Unavailable`；候选数与 decision 批次对账；review 不产生订单或配置写入；Champion/Challenger 使用同一输入时间；未满足样本策略不能晋升；一次批准的提案仍可通过新版本和回滚版本完全追溯。

## 6. 实施顺序与模块边界检查

先实现模块一的 envelope/health authority，再实现模块二的记录；没有这两者，纸面订单没有可复现输入，模块三不能开始。模块三交付可重放和 WORM 回执后，模块四才计算真实的成本后结果。每个模块完成前都要做一次边界检查：调用方是否只走公开 interface、是否把 adapter 细节泄漏到 `main.rs`、是否有非 `Available` 数据被转成数值、是否有无审计回执的状态改变。

该顺序复用而非推翻现有 `data_provider`、`strategy`、`risk`、`paper_trade`、`portfolio` 与 push 模块；它们逐步移到新 seam 后，旧路径只读或由 feature flag 关闭。任何实盘 broker adapter 仍处于本设计之外，必须另起 Gate L 设计与 PR。
