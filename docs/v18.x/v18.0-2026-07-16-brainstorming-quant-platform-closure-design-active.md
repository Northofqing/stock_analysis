## 1. 定位与核心结论

### 1.1 当前定位

项目已经具备可信的 A 股事件驱动监控、多源数据、信号发现、模拟交易和复盘基础，但尚不是可审计、可复现的研究到实盘交易系统。

当前产品链路在“建议”与“可验证经济结果”之间断裂：数据、模型、决策、模拟成交、账本和复盘尚未由统一的不可变关联键与审计凭证贯通。

v18.0 的首要目标不是增加更多策略，而是先回答以下问题：

1. 哪一份真实数据、在什么时点、以什么质量状态生成了信号？
2. 哪个模型、配置和组合约束批准或拒绝了决策？
3. 在明确的成交与成本模型下，本应发生什么？
4. 结果支持、证伪或限制了哪个假设？
5. 若未来启用实盘，券商真相是否与内部订单、成交、资金和持仓完全对账？

### 1.2 两阶段产品路线

v18 分为两个不能倒置的阶段：

1. **机构级模拟闭环：** 可复现数据与研究、成本后组合决策、统一风控、持久化订单/成交/账本，以及可证伪复盘。
2. **受控实盘门禁：** 仅在模拟闭环积累稳定证据后，另行建设券商生命周期、对账、运维控制和人工审批。

在 Gate L 通过前，系统只能被表述为研究、监控、通知和模拟交易平台。推送成功、人工调整持仓或模拟成交都不得表述为券商确认的真实成交。

## 2. 评估基线与当前能力

### 2.1 证据边界

2026-07-16 的来源评估检查了生产 monitor 集成面、数据源、质量与时效、交易与风控、数据库、回测、复盘和版本化设计。

当时 `cargo test --lib` 的结果为 1,164 passed、0 failed、7 ignored，但仍有未使用代码、弃用接口和不可达分支等警告。因此该结果只能证明单元测试基础较广，不能证明满足 `clippy -D warnings` 或发布就绪。

`AGENTS.md` 要求读取的 `.github/copilot-instructions.md` 在评估时及本次整合时均不存在。本文不推断其内容；该缺失应在实际实施 PR 的 pre-flight 中继续披露。

### 2.2 能力地图

| 能力 | 现有基础 | v18 判断 |
| --- | --- | --- |
| 多源日线 | `data_provider/fallback.rs` 会验证候选日线 | 基础较强，应保留质量门禁并统一来源证据 |
| 数据质量 | `monitor/data_quality.rs`、freshness 相关模块 | 控制已存在，但还不是所有行动路径的唯一权威 |
| 事件与推送 | `push_l1`、`push_l4`–`push_l7`、v17 集成 | 继续作为通知传输；不得承担交易确认语义 |
| 信号与机会发现 | `monitor/`、`opportunity/`、`market_analyzer/`、`strategy/` | 能力广，但来源证据尚未统一传播到结果 |
| 回测研究 | `pipeline/backtest_runner.rs`、`strategy/core.rs` | 可复用；缺 point-in-time 快照时的结果只能标为探索性 |
| 组合与风控 | `portfolio/`、`risk/`、`trading/risk_adapter.rs` | 已有硬规则基础，需要收敛为单一策略入口 |
| 模拟交易 | `trading/paper_trade.rs`、`paper_engine.rs` | 已与真实持仓分离；仍缺完整 fill、账本和归因契约 |
| 实盘执行 | 已验证的是 `SimulatedExecutionGateway` | 未证明具体券商写入端到端能力 |
| 执行对账 | `execution_tracking.rs` 主要是内存报告模型 | 尚无持久化券商/资金/持仓对账闭环 |
| 审计治理 | 已有账户模式与推送分析存储 | 尚无贯穿决策到成交的不可篡改审计真相源 |

### 2.3 必须保留的优势

- 日线 fallback 只接受通过质量验证的数据；全部来源失败时显式报错，符合 2.1 与 2.3。
- 模拟交易写入 `paper_trades`，不混入真实 `stock_position`，方向上符合 2.5。
- 已存在手数、涨跌停、资金/持仓、账户模式和业务订单去重等 A 股安全概念。
- 因子快照、滚动验证、数据时效和盘后复盘已被视为一等问题，不应另起重复体系。

| P1 | 数据、模型、决策、模拟成交和复盘缺共同键 | 结果不可解释、不可证伪 | 2.7 | 全链路使用 `correlation_id` 与不可变 `decision_id` |

## 4. 目标、非目标与安全边界

### 4.1 目标

- 每个行动输入都有来源、语义时间、schema、调整口径、质量状态与验证证据。
- 每次候选评估和组合意图固化为不可变 `DecisionRecord`。
- 每次批准与拒绝都经过统一风控，并持久化为可审计结果。
- 模拟订单、成交、费用、资金和持仓可由事件完整重放与对账。
- 复盘沿 `decision_id` 追溯实际成本后结果，并只提出受治理的变更建议。

### 4.2 初始版本非目标

- 在现有证据可信前继续扩张 alpha 策略数量。
- 把通知送达当作交易执行。
- 用默认值、空集合或合成状态替代缺失的真实数据。
- 在券商对账、kill switch 和运维门禁完成前启用实盘写路径。
- 让复盘模块直接改写模型、配置或风险阈值。

### 4.3 不可妥协的红线映射

| 规则 | v18 落点 |
| --- | --- |
| 2.1、2.2 | 数据失败统一为不可行动状态，不允许默认值或假数据兜底 |
| 2.3 | 价格、连续性、重复、异常跳变和复权一致性进入 validation evidence |
| 2.4 | 行情 5 秒、账户 30 秒、日线 1 个交易日由单一 freshness policy 执行 |
| 2.5 | 模拟账本与真实账户物理、命名空间和写入路径隔离 |
| 2.6 | A 股硬限制、幂等与二次确认由统一风控执行 |
| 2.7 | 关键事件先获得至少五年不可变保留的审计回执 |
| 2.8 | save/verify/notify/reconcile 等接口必须真实操作目标系统 |
| 2.9 | 阈值或配置变更必须有 spec 与配置字段双向证据 |
| 2.10 | 过滤、排序、去重、互斥和限额先登记业务规则 ID |

## 5. 目标架构与数据流

```text
真实数据源
  │ 来源时间、schema、质量证据
  ▼
Data Contract Gate ──拒绝/记录──> Data Incident Ledger
  │ Available、fresh、valid 的 DataHealthSnapshot
  ▼
Research & Model Registry ──> Signal / Feature Snapshot
  │ 不可变模型、配置与数据版本
  ▼
Decision & Portfolio Constructor ──> DecisionRecord
  │ 预期收益、不确定性、流动性、成本、风险预算
  ▼
Unified Risk Policy ──拒绝/记录──> RiskDecisionRecord
  │ 模拟模式只生成 parent paper order
  ▼
Paper Execution & Fill Model ──> PaperOrder / PaperFill / PaperLedger
  │ 涨跌停、T+1、费用、滑点、部分成交与未成交
  ▼
Outcome, Attribution & Review ──> 模型监控与晋升证据

未来且仅位于 Gate L 之后：
ApprovedOrder → Broker Adapter → BrokerOrder/Fills → Reconciliation → Audit Journal
```

每条边都携带 `correlation_id`。`decision_id` 一经生成不可变，并连接输入快照、模型版本、组合构造、风控结果、模拟/实盘订单、成交、复盘和告警。

### 5.1 四模块依赖

```text
现有数据适配器 ───────────────┐
                              ▼
                    ① DataHealthSnapshot
                              │ envelope + snapshot_id
现有策略 ─────────────────────┼────────────┐
组合/风控上下文 ──────────────┘            ▼
                                    ② DecisionRecord
                                             │ approved paper intent
                                             ▼
                              ③ Paper Order / Fill / Ledger
                                             │ immutable event refs
                                             ▼
                           ④ Attribution / ModelChangeProposal
                                             │ reviewed proposal only
                                             └── 下一轮 Champion/Challenger
```

依赖顺序不可跳过。没有模块一和模块二，就不存在可复现的模拟订单输入；模块三未交付可重放账本与审计回执，模块四就不能声称计算了真实的成本后结果。

## 6. 共同领域契约

### 6.1 数据状态

每个行动输入使用 `DataEnvelope<T>`，至少包含：

| 字段 | 含义 |
| --- | --- |
| `source_id`、`source_timestamp`、`received_at` | 来源与延迟 |
| `instrument`、`market_session`、`as_of` | 语义时间与适用范围 |
| `schema_version`、`adjustment_basis` | 解释版本与复权口径 |
| `availability` | `Available`、`Unavailable`、`Stale`、`Invalid`、`Conflicted` |
| `validation_evidence` | 价格、连续性、跳变、重复与公司行动检查 |

只有 `Available` 可以携带 `Some(value)`。其余状态必须携带 `None` 和可诊断原因，不能被解释为零、空集合或正常状态。

业务字段缺失与数据源失败必须分开表达。某财报中合法缺少字段，可在 `Available` 载荷中保留 `None` 并告警；数据源失败必须是 `Unavailable`。

### 6.2 身份与不可变性

- `correlation_id` 标识一次端到端评估。
- `decision_id` 是后续订单、成交和复盘的稳定连接键。
- `candidate_batch_id` 标识候选全集及其版本组合。
- 新行情、账户、模型或配置必须产生新记录，不能覆盖旧决策。
- 通知重试只能重新渲染已持久化决策，不能重做决策。

### 6.3 审计 seam

四个模块只依赖窄 `AuditJournal` 接口，不依赖具体对象存储或 SQLite 实现。关键事件的 canonical payload 必须使用固定 schema、稳定字段顺序和 UTC 时间。

最小写入顺序如下：

1. 生成 canonical event。
2. 写入启用 WORM/Object-Lock 的不可变对象，取得版本化 `AuditReceipt`。
3. 在同一 SQLite transaction 中写领域事件和回执引用。
4. 更新可重建的查询投影。

第二步失败时，模拟订单提交必须 fail-closed。第三步失败时，不可变对象成为“归档孤儿”，由日终对账识别并以纠正事件关闭，不能静默丢弃。

每天应计算签名 Merkle root，并写入另一份受保留锁保护的对象。SQLite 是运行投影和查询缓存，不是审计真相源；单独哈希链、普通本地文件或 append-only SQLite 均不满足 2.7。

## 7. 模块一：Data Contract Gate / DataHealthSnapshot

### 7.1 职责

建议位于 `src/data_contract/`。模块把 provider、质量校验、数据模式和 freshness 收敛为行动前的唯一数据事实，但不负责 HTTP 拉取，也不判断投资信号。

公开接口只需支持两类能力：

- `assess(input, now) -> DataHealthSnapshot`：从真实来源时间和验证证据生成快照。
- `check(snapshot, intent) -> Result`：按行动类型验证必需能力是否可用且新鲜。

### 7.2 不变量

- provider、网络、授权或无结果统一为 `Unavailable { reason, retryable }`。
- 生产代码不得直接构造 `Full`；测试 fixture 必须有明确测试命名并被生产 guard 拒绝。
- 开仓、加仓、调仓所需能力必须全部 `Available` 且未过期。
- 行情阈值为 5 秒、账户为 30 秒、日线按交易日规则判断。
- 减仓或平仓是否在数据退化时允许，由版本化 `ActionIntent` 和风控明确决定，不在本模块暗中放行。

### 7.3 现有模块适配

| 现有模块 | 采用方式 | 必须删除或隔离的行为 |
| --- | --- | --- |
| `data_provider/fallback.rs` | 包装已验证日线，并保留全部来源观察引用 | 最终错误转为空数据继续计算 |
| `monitor/data_quality.rs` | 生成价格、连续性、复权和重复证据 | 只记录日志后仍交给策略 |
| `data_provider/service.rs` | 财务、资金流、盘中形态返回 envelope/result | `unwrap_or_default` 形成可行动零值 |
| `monitor/data_mode.rs`、freshness 路径 | 成为 health authority 的输入适配器 | 生产路径直接构造 `DataMode::Full` |

建议投影表为 `data_health_snapshots`、`data_capability_health`、`data_incidents`。表中只存状态、来源引用和回执，不复制大型行情载荷。

### 7.4 最小验收

- 6 秒行情拒绝开仓。
- 31 秒账户快照拒绝加仓。
- provider 超时得到 `Unavailable`，不等于零资金流。
- 价格 ≤0 或重复时间得到 `Invalid`。
- 多源冲突得到 `Conflicted`。
- 非交易日跨度按交易日历判断，不误判日线过期。
- `DataHealthRejection` 不能被风控或模拟执行绕过。

## 8. 模块二：CandidateSnapshot / DecisionRecord

### 8.1 职责

建议位于 `src/decision/`。模块固化一次完整候选评估与组合意图，不发订单、不推送，也不对候选集做隐式截断。

候选全集是回答“当时为什么没有选择另一只股票”的必要证据。入选与未入选标的都要保留 disposition、规则 ID、原因、特征引用、信号证据、预期收益、不确定性、流动性和成本估计。

### 8.2 决策身份与幂等

候选批次身份至少由 strategy、model、config、universe、data-health 和 portfolio revision 组成。

完全相同的提交返回已持久化 `DecisionRecord`；同一批次键出现不同载荷时属于完整性冲突，必须拒绝并审计，不能覆盖。

组合构造输出目标仓位、差额、成本、容量、约束证据以及 `Proposed` 或 `Rejected`。缺少成本或流动性证据时必须拒绝，不能假设零成本。

### 8.3 模块边界

- `strategy/core.rs` 适配为 `SignalEvidence`。
- `pipeline/backtest_runner.rs` 只产生通过 point-in-time 检查的快照引用。
- `portfolio/*` 适配为组合构造输入。
- `push_l1`–`push_l7` 只消费已持久化的 decision view。
- `main.rs` 只组织 adapter，不自行计算或生成 `decision_id`。

建议投影表为 `candidate_batches`、`candidate_evaluations`、`decision_records`、`decision_constraints`。

### 8.4 最小验收

- 同输入重试只得到同一决策身份。
- 候选数量能与 universe 快照对账。
- 过滤、排序、容量限制缺少 BR ID 时拒绝构造。
- 任一必需特征不是 `Available` 时，不产生 `ApprovedPaperIntent`。
- `DecisionRecord` 取得审计回执后不可变。

## 9. 研究与模型注册

研究记录至少包含数据集版本、universe、交易日历、复权策略、特征快照完整性、模型/配置 hash、代码 revision、成本假设和指标。

模型进入模拟交易前必须满足：

- 评估期内 point-in-time 因子与信号完整。
- 样本外与 walk-forward 测试通过明确门槛。
- 报告成本后收益、换手、容量/流动性、回撤和因子/行业暴露。
- 独立评审记录局限性、批准/拒绝、有效期和复审日期。

使用现值填补历史特征缺口的结果只能标为“探索性”，不得用于模型晋升。

## 10. 统一风控策略

每个模拟订单以及未来每个券商订单都必须通过同一策略接口。策略至少执行：

- 测试/实盘标的命名空间与账户隔离（2.5）。
- 可用资金、单笔 100 万元上限、100 股整数倍、涨跌停、60 秒同业务订单号去重和 50 万元二次确认（2.6）。
- 行情与账户快照真实、可用且新鲜。
- 模拟模式只能生成模拟订单；实盘模式还需账户启用、操作员授权、必要的双人确认与 kill switch 检查。

批准与拒绝都持久化为 `RiskDecisionRecord`。只写日志而不留目标数据记录属于 2.8 禁止的假实现。

## 11. 模块三：Paper Order / Fill / Ledger

### 11.1 职责与状态机

建议位于 `src/paper_ledger/`。模块把一次性模拟记录升级为可重放事件模型，同时继续与真实持仓和未来 broker adapter 物理隔离。

```text
Proposed → RiskApproved → Submitted → PartiallyFilled → Filled
                         │                │
                         ├─ Rejected      ├─ Cancelled
                         │                ├─ Expired / NotFilled
                         │                └─ ReconciliationBlocked
                         └─ AuditUnavailable
```

`RiskApproved` 只表示统一风控已通过，不是交易所状态。进入 `Submitted` 前必须已有决策与风控两份审计回执；每个 fill 也有独立数据快照、成本模型版本和回执。

### 11.2 幂等与成交模型

模拟订单键采用已登记规则 BR-040：`decision_id + target_revision + side`。重试返回既有 receipt 或明确失败，不能重复扣款或新增虚拟持仓。

只有 `process_market` 可以创建 fill。版本化 `FillModel` 必须处理涨跌停、T+1、参与率、盘口/成交量、佣金、印花税、滑点、部分成交和未成交。

模型缺少参数时只能产生 `NotFilled` 或错误，不得假定理想成交。

### 11.3 账本与对账

`PaperLedger` 由事件重放生成现金、冻结资金、持仓、可卖量、费用和已/未实现 P&L。汇总投影可删除重建，事件和审计对象不可删除。

建议表为 `paper_orders`、`paper_events`、`paper_fills`、`paper_ledger_projection`、`paper_reconciliations`。真实 `database/positions.rs` 只可作为风控输入，禁止写入模拟结果。

日终对账从事件重放，并比较订单状态、现金、持仓、可卖量、费用、审计链与每日根。任何差异、孤儿对象或缺回执都要建立 incident，并阻断下一笔模拟订单。

纠正只能追加 `CorrectionRecorded`，保留原事件。不得 UPDATE 汇总表来“修平”差异。

### 11.4 最小验收

- 涨停买入形成持久化 `NotFilled`。
- T+1 非法卖出被拒绝。
- 部分成交和费用计算正确。
- 相同订单键重试不重复。
- 事件重放结果与投影一致。
- 少于五年保留锁或缺少审计回执时拒绝提交。
- 对账差异阻断下一笔订单。

## 12. 模块四：Attribution / ModelChangeProposal

### 12.1 职责

建议位于 `src/review/`，指标计算可位于 `src/performance/`。模块将“策略看起来有效”转化为可复核、可证伪且不能越权改参数的结果。

复盘只沿以下链路读取：

```text
decision_id → risk result → paper order → fills → ledger → market outcome
```

复盘同时保留入选与未入选候选，按模型版本、持有周期、市场状态、行业/风格、rank、流动性和拒绝原因分组。

T+1、T+5、T+20、退出时点和最大有利/不利变动等标签若观察窗口不足，必须保持 `Unavailable`，不能补零。

### 12.2 最小指标

- 毛/净 P&L 与相对基准收益。
- 命中率、收益分布、换手与胜率校准。
- 预测成本与实际成本差异。
- fill rate、implementation shortfall 与参与率。
- 集中度、行业/风格暴露与回撤。
- 信号、组合构造、执行成本和市场 beta 的分项归因。
- 被证伪或受限制的假设。

成本后的失败不能自动归为模型预测错误，必须区分信号、组合构造与执行影响。

### 12.3 变更治理

```text
AttributionReport
      │ evidence + sample policy
      ▼
ModelChangeProposal(Draft) → Reviewed → Shadow → PaperChallenger
                                  │                  │
                                  ├─ Rejected         ├─ Promoted（人工批准）
                                  └─ NeedsMoreData    └─ Restricted / Retired
```

提案可以建议修改模型、特征、风险预算或成交假设，但不能直接写 `.toml`。生效必须通过独立 PR、版本化发布、阈值证明和回滚版本。

Champion 与 Challenger 必须消费相同 universe、数据健康快照、行情时间、账户快照和 fill/cost 假设，并分别维护 `paper_book_id`。

晋升不能只依赖夏普或胜率。样本策略还要检查持有期成熟度、市场状态覆盖、成本后结果、尾部风险、数据可用率、多重比较说明和经济容量；任一缺失只能进入 `NeedsMoreData`。

### 12.4 最小验收

- 同输入产生相同 report hash。
- 缺失链路显示 `Unavailable`。
- 候选数与决策批次可对账。
- review 不产生订单或配置写入。
- Champion/Challenger 使用相同输入时点。
- 样本策略不满足时无法晋升。
- 已批准提案仍可追溯新版本和回滚版本。

## 13. 失败模式与恢复

| 失败 | 必需响应 | 恢复或回滚 |
| --- | --- | --- |
| 行情/账户缺失或过期 | 拒绝开仓或增加风险；记录 incident | 刷新真实数据并重建快照，禁止默认值 |
| 数据质量或公司行动冲突 | 隔离标的或序列 | 人工确认并生成修正版数据 |
| 多源意见不一致 | 标记 `Conflicted`，停止行动 | 保留两边观察，按已登记规则解决 |
| 模拟账本不变量破坏 | 停止模拟执行，保留不可变记录 | 从事件重放并修复根因 |
| 券商不可用或确认含糊 | 未来 adapter 进入 `Unknown`，阻断关联订单 | 查询券商真相后对账，禁止推测成交 |
| kill switch、账户模式或风控违规 | 拒绝并持久化原因 | 仅允许经授权、可审计的恢复 |
| 模型监控越界 | 停止晋升并限制模型 | 独立复核、新版本或退役 |
| 审计归档不可用 | 不接受关键状态改变 | 恢复不可变存储并完成校验后再开 |

## 15. 分阶段实施路线

### 15.1 顺序与依赖

| 顺序 | 工作流 | 依赖 | 退出条件 |
| ---: | --- | --- | --- |
| 1 | 数据契约与健康权威 | 无 | 默认/假数据不能进入决策，时效阈值生效 |
| 2 | 研究与模型注册 | 1 | point-in-time 与成本后证据成为晋升条件 |
| 3 | 决策、组合与统一风控 | 1、2 | 每个模拟行动都有持久化决策与风控结果 |
| 4 | 模拟订单、成交、账本与对账 | 1、3 | 重放和对账不变量测试通过 |
| 5 | 结果归因与运营复盘 | 2、4 | 结果可以追溯并证伪决策 |
| 6 | 实盘就绪评审 | 1–5 | 仅产出 Gate L 设计决定，不写券商订单代码 |

每个工作流必须单独形成小 PR、设计更新、失败路径测试、Gate 证据和回滚说明。前一工作流的 Gate 未关闭时，不得启动下一工作流。

### 15.2 工作流一：数据契约

1. 先写测试，证明 provider 失败只能形成 `Unavailable`。
2. 引入带来源、时间、复权、状态和证据的 envelope。
3. 清除决策路径的默认值/空值兜底。
4. 建立唯一 `DataHealthSnapshot`，执行 5 秒和 30 秒要求。
5. 删除或隔离生产路径的合成 `Full` 构造器。
6. 为新增过滤、去重或限速登记 BR，再运行 Gate B/C。

### 15.3 工作流二：研究与模型注册

1. 记录代码、universe、日历、数据、复权、配置和成本版本。
2. promotable 回测必须具备 point-in-time 因子。
3. 生成样本外/walk-forward 与成本后报告。
4. 增加独立评审、局限、有效期和监控阈值。
5. 验证数据或配置变化会产生新研究身份，不能覆盖批准结果。

### 15.4 工作流三：决策与统一风控

1. 定义稳定的 `DecisionRecord` 和 `RiskDecisionRecord`。
2. 组合构造纳入净预期、容量、换手、现金、集中度、暴露和回撤。
3. 把现有模拟风控合并到一个策略接口。
4. 执行数据健康、账户隔离、A 股硬限制、去重与二次确认。
5. 在任何模拟执行副作用前持久化接受或拒绝结果。

### 15.5 工作流四：模拟执行与账本

1. 保留模拟/实盘隔离，引入 parent order、child fill 与完整状态机。
2. 版本化成交和成本假设，并记录每个 fill 的行情证据。
3. 建立事件支持的模拟账本，SQLite 汇总仅作投影。
4. 关键事件先取得五年 WORM 回执，再接受状态改变。
5. 实现日终不变量、回执、每日根和归档孤儿对账。
6. 执行保留策略探针及 restore/read drill。
7. 旧模拟记录只读迁移，不重解释为新模型 fill。

### 15.6 工作流五：归因与复盘

1. 用 `decision_id` 关联的持久化结果替代内存执行统计。
2. 报告成本差、成交率、implementation shortfall、净绩效、校准、暴露和回撤。
3. 增加受治理的限制/晋升决定，复盘不能直接改配置。
4. 通知只渲染已持久化报告。

### 15.7 工作流六：实盘就绪评审

1. 审核约定模拟观察期内的 Gate P 证据。
2. 单独设计券商状态机、查询、对账、kill switch、授权和双人确认。
3. 桌面演练含糊确认、券商中断、重启、重复提交、资金/持仓不一致和 kill switch 恢复。
4. 获取单独的实盘设计批准；否则继续保持模拟模式。

## 16. 发布门禁

### 16.1 Gate P：机构级模拟闭环

Gate P 通过必须同时满足：

- 所有决策都能追溯到真实来源、版本和时间，必需字段不来自默认值。
- 每个关键事件都有经验证的不可变回执，WORM/Object-Lock 保留至少五年。
- 每日根重放和 restore/read drill 通过。
- 模拟账本每日对账，未解释差异为零。
- 研究晋升具备 point-in-time、样本外/walk-forward 和成本后证据。
- 过期、缺失、坏数据、冲突数据和风控拒绝的关键失败测试通过。
- 仓库规定的 fmt、clippy、test 和 compliance 全部通过。

### 16.2 Gate L：受控实盘执行

Gate L 默认关闭。只有 Gate P 在约定模拟期内稳定，并同时满足以下条件，才可另行申请打开：

- 券商 adapter 使用幂等业务订单号和持久化订单状态机。
- 确认、成交、撤单、资金和持仓能在盘中及日终对账。
- 任何 `Unknown` 状态都会停止交易。
- UI、推送和重试路径都不能绕过本地预交易控制。
- 账户隔离、操作员授权、双人确认、kill switch、审计保留和 runbook 已被独立测试。
- 经明确批准的小资金 canary 验证完整生命周期后，才可讨论扩容。

## 17. 推进、验证与回滚

### 17.1 渐进接入

1. 数据与决策契约先以 shadow 模式接入，只比较既有 monitor 输出。
2. 再把选定策略接入新模拟账本；旧模拟记录只读保留。
3. Gate P 证据完整后才晋升，所有推送继续保持信息性。
4. Gate L 前置条件满足后，券商 adapter 仍需单独设计和 PR。

feature flag 的默认值必须可见且安全，不得静默启用执行。回滚关闭新路由，保留所有 append-only 记录，并恢复到之前的监控与推送路径。

数据或风控违规不是简单关闭代码即可结束，必须按 `AGENTS.md` §3.2 回到 Gate A 复核数据流和失败模式。

### 17.2 每个实施 PR 的最小证据

- `Refs: spec §<本文节号>`。
- 按影响列出 `Data-Redlines`。
- 列出旧模块 adopt/reject/change 及原因。
- 阈值/配置变更提供 `Threshold-Proof`。
- 过滤、排序、去重、互斥和限额列出 BR ID。
- 列出失败路径测试名与事件/审计样例。
- 提供验证输出和 `git revert <commit>` 回滚步骤。

### 17.3 全局验证命令

```bash
cargo fmt --check
cargo clippy -- -D warnings
cargo test
bash tools/compliance/check.sh
git diff --check
```

数据库或事件迁移还必须提供向前兼容与 restore/replay 方案。回滚不得删除审计、订单、成交或模拟账本记录。
