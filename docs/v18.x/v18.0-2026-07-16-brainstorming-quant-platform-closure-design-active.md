# v18.0 Quant Platform Closure Design

**Status:** active design, approved for planning
**Scope:** research-to-paper-trading closure first; live execution is disabled until the explicit Live Execution Gate passes.
**Refs:** `AGENTS.md` §§2.1–2.10; `docs/ENGINEERING_RULES_V2.md` §§1–2; `v18.0-2026-07-16-review-quant-platform-assessment.md`.

## 1. Intent and non-goals

v18 turns the monitor into a system that can make reproducible, cost-aware, falsifiable paper-trading decisions. It does not claim or enable autonomous live trading.

Non-goals for the initial v18 release:

- adding more alpha strategies before current evidence is trustworthy;
- treating notification delivery as trade execution;
- silently substituting absent market/account data;
- enabling a broker write path before reconciliation and operational gates are complete.

## 2. Target architecture

```text
real sources
  │ source timestamp, schema version, quality evidence
  ▼
Data Contract Gate ──reject/record──> Data Incident Ledger
  │ Available + fresh + valid DataHealthSnapshot
  ▼
Research & Model Registry ──> Signal / Feature Snapshot
  │ immutable model/config/data versions
  ▼
Decision & Portfolio Constructor ──> DecisionRecord
  │ expected return, uncertainty, liquidity, cost, risk budget
  ▼
Risk Policy Gate ──reject/record──> RiskDecisionRecord
  │ paper mode: creates parent paper order only
  ▼
Paper Execution & Fill Model ──> PaperOrder / PaperFill / PaperLedger
  │ quotes, limits, T+1, fees, slippage, partial/not-filled outcomes
  ▼
Outcome, Attribution & Review ──> model monitoring / promotion evidence

Future only, behind Live Execution Gate:
ApprovedOrder → Broker Adapter → BrokerOrder/Fills → Reconciliation → immutable audit ledger
```

Each arrow carries `correlation_id`; `decision_id` is immutable and joins input snapshots, model version, portfolio construction, policy result, paper/broker order, fills, review, and alerts.

## 3. Domain contracts

### 3.1 Data Contract Gate

Every input used for a decision has a typed envelope:

| Field | Meaning |
| --- | --- |
| `source_id`, `source_timestamp`, `received_at` | provenance and latency |
| `instrument`, `market_session`, `as_of` | semantic time and scope |
| `schema_version`, `adjustment_basis` | interpretation and corporate-action basis |
| `quality_status` | `Available`, `Unavailable`, `Stale`, `Invalid`, or `Conflicted` |
| `validation_evidence` | price, continuity, jump, duplicate, and corporate-action checks |

Rules:

- Provider failures have the single representation `Unavailable { reason, retryable }`; they never become numeric defaults in decision inputs (2.1, 2.2). `Unavailable` envelopes carry no value.
- Quotes used to increase/open a position must be ≤5 seconds old; account cash/position must be ≤30 seconds old; daily/history must meet the trading-day rule (2.4).
- Invalid continuity, duplicate time, price ≤0, abnormal jump without approval, or inconsistent adjustment basis rejects the input (2.3).
- A single process-owned `DataHealthSnapshot` is built from source timestamps. Production code cannot construct `Full` directly; test helpers are separately named and rejected by production guards.

### 3.2 Research and Model Registry

A research run records dataset version, universe, trading calendar, adjustment policy, feature snapshot availability, model/config hash, code revision, transaction-cost assumptions, and metrics. A model may enter paper trading only after:

- point-in-time factor/signal data are complete for the evaluated period;
- out-of-sample and walk-forward tests pass stated acceptance thresholds;
- net-cost return, turnover, capacity/liquidity, drawdown, and factor/sector exposures are reported;
- an independent review records limitations and an expiry/review date.

Results that use present-day values to fill historical feature gaps are labelled **exploratory** and cannot promote a model.

### 3.3 Decision and portfolio construction

`DecisionRecord` is the sole intent artifact. It contains eligible universe result, signal evidence, expected return and uncertainty, intended horizon, liquidity/capacity estimate, expected costs, portfolio target, constraints, and model/config/data hashes.

The constructor optimizes expected **net** value subject to cash, sector/industry, single-name, turnover, liquidity participation, drawdown, factor, and concentration budgets. It returns a proposed parent order or a deterministic rejection. Ranking, filtering, limits, and deduplication introduced here must first be registered in `docs/business_rules.md` (2.10).

### 3.4 Unified risk policy

One policy gateway runs before every paper order and, later, every broker order. It enforces the existing hard rules plus the new data contract:

- test/live namespace and account isolation (2.5);
- cash, RMB order cap, 100-share lot, price-limit, duplicate business ID, and secondary confirmation (2.6);
- fresh available quote/account data; no action on `Unavailable`, `Stale`, `Invalid`, or `Conflicted` data;
- mode-specific authority: paper mode may simulate; broker mode requires an enabled account, authorized operator, secondary confirmation when required, and an active kill-switch check.

A rejection is a durable `RiskDecisionRecord`, never merely a log line.

### 3.5 Tamper-resistant audit journal

SQLite/Diesel is an operational projection and query cache, **not** the audit system of record. Every critical data acceptance/rejection, decision, risk result, paper order, fill, cancellation, reconciliation and model-governance event is canonicalized and must receive an immutable `AuditReceipt` before the corresponding action is accepted.

The minimum v18 adapter writes one immutable object per critical event to storage that enforces WORM/Object-Lock retention of at least five years. The object contains its canonical payload hash, predecessor hash, event hash, event time, and retention-until time; the receipt contains immutable object version/ETag, chain hash, archive time, and retention-until time. A daily signed Merkle root, stored in a separately retained object, makes deletion/reordering detectable. A failed archive write is fail-closed for paper-order submission; an archive object that succeeds but whose SQLite transaction fails is detected as an orphan during reconciliation, never silently discarded.

Gate P evidence must include a retention-policy probe showing compliance/WORM mode and five-year retention, a replay that verifies every receipt and daily root for the paper period, and a restore/read drill. Hash chaining alone and an application-level append-only SQLite table do not satisfy AGENTS.md §2.7.

### 3.6 Paper execution, cost, and ledger

Paper execution models parent orders, child fills, no-fill/partial-fill states, exchange limits, T+1 availability, commissions, taxes, spread/slippage, and explicit fill assumptions. It does not write real positions.

`PaperLedger` records cash, holdings, realized/unrealized P&L, fees, and mark source. Each paper fill preserves the market data version and cost model version used. Daily reconciliation checks ledger invariants; discrepancies halt subsequent paper orders and open an incident.

### 3.7 Outcomes and review

Review consumes `DecisionRecord` and actual paper outcomes, not push counts. It reports gross/net performance, fill rate, implementation shortfall, realized costs versus estimates, calibration, hit rate by regime, exposure, drawdown, and hypothesis failures. The review proposes, but does not directly apply, model/config changes; promotion requires recorded approval and a new model/config version.

### 3.8 Implementation-level companion

The module interfaces, state machines, storage boundaries, adapter points, and test seams are normative in [the four-core-module design](v18.0-2026-07-16-codebase-design-four-core-modules.md). It is the implementation companion to this active architecture; any intentional deviation requires an ADR and a matching update here.

## 4. Failure modes

| Failure | Required response | Recovery / rollback |
| --- | --- | --- |
| quote/account data stale or missing | reject new/increasing paper or broker order; emit incident | refresh real source; rebuild snapshot; never use default value |
| data quality/corporate-action conflict | quarantine affected instrument/series | manual confirmation and corrected source version |
| source disagreement | mark `Conflicted`; no action until resolution policy selects evidence | investigate source and retain both observations |
| paper ledger invariant mismatch | halt paper execution, preserve immutable records | reconcile from append-only events; fix root cause before resume |
| broker unavailable or acknowledgement ambiguous | future broker adapter enters `Unknown`; block dependent orders | query broker truth and reconcile; never infer fill |
| kill switch / account mode / risk breach | reject order and retain reason | authorized, audited reset only |
| model monitoring breach | stop promotion and mark model restricted | independent review, new version, or retirement |

| giant `main.rs` / `push_templates.rs` paths | narrow | use adapters; do not add v18 business logic directly to these integration files |

## 6. Release gates

### Gate P — Institutional paper loop

- all decisions trace to real source/version/time and no required field comes from a default fallback;
- every critical paper-loop event has a verified immutable audit receipt with WORM/Object-Lock retention of at least five years; daily root replay and restore/read drill pass;
- paper ledger reconciles daily to its own event/fill records with zero unexplained breaks;
- research promotions show point-in-time data, out-of-sample/walk-forward evidence, and net-cost metrics;
- critical failure tests cover stale/missing/bad/conflicted data and risk-policy rejection;
- `cargo fmt --check`, `cargo clippy -- -D warnings`, `cargo test`, and `bash tools/compliance/check.sh` pass.

### Gate L — Controlled live execution (future)

Gate L is closed by default. It opens only after Gate P is stable for a defined paper period and all of the following exist:

- broker adapter uses idempotent business order IDs and a durable order state machine;
- broker acknowledgements, fills, cancels, cash, and positions reconcile intraday and end-of-day; unknown states stop trading;
- pre-trade controls are locally enforced and cannot be bypassed by UI, push, or retry path;
- account/test isolation, operator authorization, two-person confirmation, kill switch, audit retention, and operational runbook have been independently tested;
- a small-capital, explicitly approved canary proves the complete lifecycle before any expansion.

## 7. Rollout and rollback

1. Introduce data and decision contracts in shadow mode; compare them against existing monitor output without changing recommendations.
2. Route selected strategies through the paper ledger; retain old paper records read-only and reconcile both paths.
3. Promote only after Gate P evidence is complete; keep all push paths informational.
4. Propose the broker adapter in a separate design and PR after Gate L prerequisites are met.

Feature flags default to the audible/safe state: no silent execution enablement. Rollback disables the new routing flag, preserves all append-only records, and returns to the prior monitoring/push behavior. A data/risk violation returns to Gate A design review, per `AGENTS.md` §3.2.
