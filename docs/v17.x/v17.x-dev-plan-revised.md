## 2. v1 §3.1 撤销的理由 (evidence)

### 2.1 v1 §3.1 原文

> "**v17.5-v17.8 实施前置条件**: `src/push_l1/event.rs` 里有 `pub enum EventKind` + 4 变体.
> 当前 0 hits (代码中是 SignalEvent/SignalSource/SignalPayload). **如果 v17.4 未合并**: v17.5-v17.8 都不能进实施. **先合 v17.4** (§4.1 EventKind 新建 + 4 种新变体)."

### 2.2 实证 — 2026-07-15 commit `be9314f` 用 SignalEvent 完成 L6 集成

| v1 假设 | 实证 (commit be9314f) |
|--------|---------------------|
| "推送链路必须经过 `pub enum EventKind`" | **否**: `src/push_l6/sink.rs:32` 强制 `pub event: SignalEvent` — 早锁死类型. 重构 EventKind 必须改 push_l6 + 所有 sink impl + main.rs 调用方, 工作量超出 v17.x 范围. |
| "跳 §3.1 必 build 失败" | **否**: `cargo build --bin monitor` 通过 + 6 个 l6_sink tests 跑过 binary. |
| "SignalEvent 体系不足支撑 L6 PushMessage" | **否**: `PushMessage.event: SignalEvent` 已绑死, 6/6 tests 包括 `make_test_event()` 用 SignalEvent::new + HoldingHealth payload 跑通. |

**结论**: v1 §3.1 是写时担心的 worst case (重构 push_l1), 实证下是 NO-OP 担忧. v2 把这撤销.
