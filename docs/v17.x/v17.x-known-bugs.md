## B01 тАФ `PushRecord::try_from` rejects `push.delivery.audit` (P0)

- **Status**: Resolved by BR-130 on 2026-07-18. `PushRecord`, `HistoryQuery` and local fixtures now consume the production `push.delivery.audit` type; corrupt delivery audit rows fail the query instead of being skipped.

- **Sources**: Internal review F1 (CONFIRMED) + Codex C-high (independent re-discovery) + Live test T3 (direct evidence)
- **Files**: `src/event/push_record.rs:78` vs `src/event/envelope.rs:159/207`
- **Symptom**: `monitor --history --success-rate` returns `RateStats { total: 0, success_rate: NaN }` despite 5126 envelopes existing for the same day.
- **Root cause**:
  - `PushDeliveryEvent::event_type()` returns `"push.delivery.audit"` (envelope.rs:159)
  - `PushRecord::try_from` (push_record.rs:78) gates on `if env.event_type != "push.delivery"` and returns `EventTypeMismatch(env.event_type.clone())`
  - `EventHistory` calls `PushRecord::try_from` for every envelope, so all real audit events are silently rejected
  - `push_record.rs:290/311` also hardcodes `"push.delivery"` in test fixtures, perpetuating the wrong event type
- **Failure scenario**: Every push through `notify.rs тЖТ publish_delivery тЖТ PushDeliveryEvent тЖТ JSONL write` produces an envelope that history queries cannot consume. The user sees `success_rate: NaN` forever, even with thousands of real deliveries.
- **Evidence**:
  ```bash
  $ ./target/release/monitor --history --success-rate --date=2026-07-17
  RateStats { total: 0, pushed: 0, deduped: 0, denied: 0, failed: 0, success_rate: NaN, ... }
  total=0 pushed=0 failed=0 success_rate=NaN%
  ```
  vs. `--history` without `--success-rate` correctly shows 3+ records with `kind=announcement_v1, outcome=Pushed, latency_ms 700-900`.
- **Fix**: Change `push_record.rs:78` gate from `"push.delivery"` to `"push.delivery.audit"`. Update line 290/311 test fixtures to match. Add integration test from `notify::deliver_and_record` тЖТ `EventHistory::query`.
- **Blocker for**: `--history --success-rate` observability, v17.3 Gate C re-verification.

## B04 тАФ Frozen transition `let _ = push_normalized_event().await` swallows errors (P0)

- **Sources**: Internal review F7 (CONFIRMED) + Codex C-high (related: BR-112 disabled)
- **Files**: `src/bin/monitor/push_templates.rs:1636`
- **Symptom**: When account transitions to Frozen, the emergency MarketActionAlert push result is silently discarded.
- **Root cause**:
  ```rust

  ```
  The `let _ =` discards any `Err` or non-`Pushed` `PushOutcome`. The user sees the existing AccountMode push, but the emergency MarketActionAlert may fail silently.
- **Failure scenario**: Real market circuit-breaker scenario тАФ `AccountMode тЖТ Frozen` triggers the emergency alert, but a sink failure (Magiclaw down, governance Deny, dedup hit) is invisible. The operator never knows the alert was lost.
- **Fix**: Replace with explicit match:

  ```rust
  match crate::v17_sources::push_normalized_event(maa_event).await {
      Ok(attempt) => log::info!("[FrozenтЖТMarketAction] pushed: kind={:?} outcome={:?}", attempt.kind, attempt.outcome),
      Err(e) => log::error!("[FrozenтЖТMarketAction] push failed: {e}"),
  }
  ```
  Also consider: if `outcome != Pushed`, emit a fallback `Emergency` push via a different channel.
- **Blocker for**: Frozen-mode emergency observability.

## B05 тАФ `routed_external_ids` filled before classify_announcement result known (P0)

- **Sources**: Internal review F3 (CONFIRMED)
- **Files**: `src/bin/monitor/main.rs:8242-8244`
- **Symptom**: In a batch where some announcements fail classify and others succeed, the failed ones are silently dropped from BOTH the v17_sources path AND the legacy path.
- **Root cause**:
  ```rust
  // Main.rs ~8242:
  for ann in &anns {  // iterates ALL announcements
      if let Some(ref eid) = ann.external_id {
          routed_external_ids.insert(eid.clone());  // marks as routed regardless of classify result
      }
  }
  ```
  The earlier loop only pushes successfully-classified events into `normalized_events`, but `routed_external_ids` is populated from the full `anns` slice. Failed-classify announcements then have `routed_external_id` set, so the legacy loop at line ~7499 sees them as "routed" and skips the legacy push тАФ but they were never pushed via v17_sources either.
- **Failure scenario**: Batch with 3 announcements where 1 fails classify. Loop populates 3 external_ids into `routed_external_ids`. Legacy loop skips all 3. The 1 failed-classify announcement is silently dropped.
- **Fix**: Populate `routed_external_ids` from the successfully-classified subset:
  ```rust
  for event in &normalized_events {  // only successful
      // ... derive eid from event
      routed_external_ids.insert(eid);
  }
  ```
- **Blocker for**: Announcements with missing title or empty external_id (Eastmoney sometimes returns no `art_code`).

## B06 тАФ `gtimg_provider.rs:388` order evidence over unauthenticated HTTP (P0 тАФ SECURITY)

- **Sources**: Codex C-high
- **Files**: `src/data_provider/gtimg_provider.rs:388`
- **Symptom**: Order execution evidence (price, daily-limit bounds, source timestamp) is fetched over plaintext HTTP.
- **Root cause**: Production quote boundary uses `http://` instead of `https://`. A network intermediary can forge a coherent fresh response that passes the local freshness checks.
- **Failure scenario**: A MITM attacker intercepts the quote request, returns a forged fresh response with manipulated price/daily-limit data. The order-safety logic accepts it as legitimate, leading to execution at attacker-controlled prices.
- **Fix**:
  1. Migrate to HTTPS or authenticated broker feed
  2. Reject plaintext providers from order-related paths (hard-coded block)
  3. Add certificate pinning for broker connections
- **Blocker for**: Production safety of any order-bearing code path.

## B07 тАФ `migrations/v12-p0-paper-and-adjust/up.sql:15-16` permits NULL fill_price on Filled (P0 тАФ FINANCIAL)

- **Sources**: Codex C-high
- **Files**: `migrations/v12-p0-paper-and-adjust/up.sql:15-16`
- **Symptom**: Trades with `status='Filled'` and `fill_price=NULL` are accepted. Read-side `COALESCE(fill_price, price)` substitutes the requested price as a fake execution cost.
- **Root cause**:
  - Schema lacks `CHECK (status != 'Filled' OR fill_price IS NOT NULL)` constraint
  - UPDATE trigger does not fire for status-only updates, so a status flip to Filled without price is allowed
  - Read path silently fabricates price via `COALESCE(fill_price, price)`, contaminating position/performance calculations
- **Failure scenario**: A Filled trade is recorded without execution price. P&L calculations use the requested price as the executed price, producing phantom gains or hiding losses.
- **Fix**:
  1. Add CHECK constraint: `status != 'Filled' OR (fill_price IS NOT NULL AND fill_price > 0)`
  2. Cover `UPDATE OF status,fill_price` with the existing trigger
  3. Validate/rebuild existing tables during migration
  4. Remove the read-side `COALESCE(fill_price, price)` substitution
- **Blocker for**: Audit-grade paper-trade correctness.

## B11 тАФ `classifier.rs:187` classify_earnings inverts Beat/Miss for negative EPS (P1)

- **Sources**: Internal review F8 (CONFIRMED)
- **Files**: `src/news/aggregator/classifier.rs:187`
- **Symptom**: Loss-making companies (actual_eps < 0) can be misclassified as EarningsBeat.
- **Root cause**:
  ```rust
  let delta_pct = (actual_eps - reference) / reference.abs() * 100.0;
  ```
  When both `actual` and `reference` are negative (e.g. actual=-0.5, reference=-1.0):
  - `delta_pct = (-0.5 - (-1.0)) / 1.0 * 100 = +50%`
  - With `beat_threshold = 10%`, this classifies as `EarningsBeat`
  - Semantically: the company went from -1.0 loss to -0.5 loss (a smaller loss), but this is NOT a "beat" in the conventional sense.
- **Failure scenario**: Loss-making companies systematically misclassified as `EarningsBeat`, triggering unwanted alerts.
- **Fix**: For negative-EPS case, compare absolute losses directly (smaller absolute loss = "less bad", not "beat"). Or restrict Beat/Miss to positive-EPS only and add a separate `EarningsImproved` / `EarningsWorsened` classification for negative cases.
- **Blocker for**: Earnings classification correctness.
